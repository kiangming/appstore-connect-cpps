Implementation notes

  • Data layer reuse. One fetchAvailabilityStatesForIaps(creds, iapIds) call per page render serves both Unit C (modal filter) and Unit D (column display). The pre-fetched Map<appleIapId, AvailabilityForIap | null> threads through the existing IapListClient prop interface — no client-side refetch on modal open.
  • Concurrency budget. Reuses the existing withConcurrency<T,R> primitive (Cycle 30) at fanout 5 — matches the Phase 1 territory-fetch ceiling and stays well under Apple's 250 req/hour cap for typical 25-item lists.
  • Filter semantics. Set mode shows items whose Apple availability is null OR territoryCount = 0; Remove mode shows items whose availability has at least one territory. The null bucket (Apple fetch failed) is excluded from both modals to avoid acting on stale state.
  • Audit reuse. Each bulk row writes AVAILABILITY_SET_ALL_TERRITORIES or AVAILABILITY_REMOVE_FROM_SALES (the Phase 1 audit types) so existing dashboards/reports surface bulk activity alongside single-item edits.
  • Fail-soft (Q-K). A per-IAP failure never cancels siblings — the orchestrator iterates to completion and the modal's progress view shows successes + failures side-by-side. Aggregate severity surfaces as a toast on close.
  • Rate-limit risk. Page render = N Apple calls. Future Phase 3 candidate: server-side cache with manual refresh, mirroring the territories-cache pattern. Tracked as Hotfix candidate.
